3 系统分析
3.1 系统总需求分析
系统需求分析是软件开发生命周期中至关重要的一环。本章旨在通过对当前家政服务行业的背景研究以及对市面上同类主流产品的深入剖析，全面、准确地梳理出“贝贝”家政服务平台的功能性与非功能性需求。通过明确关键利益相关者（用户、家政人员、管理员）的业务痛点与操作期望，确立项目的设计基准，为后续的系统设计、开发、测试和验收提供客观依据。
为了确保系统的设计能够切中用户痛点并符合市场趋势，本项目主要采用了文献研究法与竞品分析法相结合的方式进行需求获取。
文献研究与行业现状分析
通过查阅近年来关于“互联网+家政”、“O2O服务模式”的相关学术论文、行业白皮书以及国家出台的相关政策文件，对家政服务市场的现状进行了系统性梳理。 研究发现，随着社会节奏加快和人口老龄化趋势加剧，家政服务需求呈井喷式增长。然而，传统家政行业长期存在“信息不对称”、“服务标准难以量化”、“售后保障缺失”等顽疾。现有文献指出，利用数字化手段构建透明、可追溯的服务平台，是解决上述问题的关键路径。这为本项目确立了“以信任和效率为核心”的设计导向。
竞品分析与现存系统痛点
本项目选取了市面上较为成熟的家政服务应用（如“58到家”、“天鹅到家”等）以及部分社区小型家政小程序进行了对比分析。通过模拟用户操作流程和分析用户在应用商店的评价反馈，总结出当前系统普遍存在的局限性：
功能臃肿与操作复杂： 大型综合平台往往集成了搬家、维修、二手交易等繁杂业务，导致APP体积庞大、菜单层级过深。对于仅有单一保洁或护理需求的用户，尤其是中老年群体，操作体验不够友好，难以快速完成下单。
个性化与社区属性缺失： 多数平台采用机械的列表展示，缺乏基于用户地理位置（LBS）的精准推荐。同时，缺乏基于社区范围的“熟人/邻里”口碑评价体系，用户在选择家政人员时往往面临“选择困难症”。
基于上述分析，“贝贝”家政服务平台将避开“大而全”的冗余设计，转而专注于打造一个轻量级、垂直化、的家政服务管理系统。
3.2 用户角色分析
贝贝家政服务系统设计了三类用户角色,分别是普通用户、工作人员和系统管理员。每类用户具有不同的功能权限和业务场景。
3.2.1 工作人员
工作人员是家政服务的提供方,即家政服务人员(阿姨)。他们通过系统接收订单、管理档期、执行服务和查看收入。
3.2.2 普通用户
普通用户是家政服务的需求方,即需要使用家政服务的个人或家庭。他们通过系统浏览服务、下单预约、管理地址和评价服务。
3.2.3 系统管理员
系统管理员是平台的运营管理方,负责系统的日常运营管理、数据监控和决策支持。管理员具有最高的系统权限,但同时也受到一定的约束(如不能禁用其他管理员)。
3.3 系统用例分析
3.3.1 系统总用例
普通用户：需要家政服务的客户，可注册登录、浏览服务、预约下单、管理订单和评价
家政阿姨：提供家政服务的工作人员，可接受订单、查看收益、管理排班
系统管理员：系统后台管理人员，负责用户、阿姨、服务、订单的全面管理

3.4 系统非功能性需求
贝贝家政服务系统在非功能性需求方面主要考虑以下几个维度。在可靠性方面，系统采用SpringBoot框架构建，具备良好的异常处理机制，能够在出现错误时及时反馈并保证核心业务不受影响，同时通过MySQL数据库的事务机制确保订单、支付等关键数据的一致性和完整性。在易用性方面，系统前端采用Vue.js框架和Element Plus组件库，提供简洁直观的用户界面，普通用户无需培训即可完成服务浏览、预约下单等操作，家政阿姨和管理员也能快速上手各自的工作台功能。在安全性方面，系统采用JWT令牌进行用户身份认证，通过Spring Security实现接口权限控制，确保不同角色只能访问其授权的功能模块，同时对用户密码进行加密存储，防止敏感信息泄露。在性能方面，系统引入Redis缓存热点数据，减少数据库访问压力，接口响应时间控制在合理范围内，能够满足中小规模家政服务平台的日常运营需求。
3.5 可行性分析
3.5.1 技术可行性
本系统采用Spring Boot + Vue.js的前后端分离架构，这是当前主流的Web应用开发模式，相关学习资源丰富，技术生态成熟。本人在本科阶段已系统学习了Java程序设计、数据库原理、Web开发等技术，具备了一定的编程基础和项目开发经验，能够理解和运用面向对象编程思想、MVC设计模式等核心概念。后端选用的Spring Boot框架提供了开箱即用的特性，大幅降低了项目配置的复杂度；My Batis-Plus简化了数据库操作；Spring Security和JWT提供了成熟的安全解决方案。前端选用的Vue.js学习曲线平缓，配合Element Plus组件库能够快速构建美观的用户界面。在开发过程中遇到的技术难点，可通过查阅官方文档、技术社区以及在指导老师的帮助下逐步解决。综上所述，本系统的技术选型合理，开发难度适中，在现有技术水平下完全能够实现预期的功能和性能要求。

3.5.2 法律可行性
本系统在设计和开发过程中充分考虑了相关法律法规的要求。在版权方面，系统采用的Spring Boot、Vue.js、MySQL、Redis等技术框架和工具均为开源软件，遵循Apache 2.0、MIT等开源协议，允许免费商业使用，不存在版权侵权风险。在隐私保护方面，系统涉及用户手机号、家庭地址等个人敏感信息的收集和存储，需遵守《中华人民共和国个人信息保护法》和《网络安全法》的相关规定。为此，系统在用户注册时明确告知信息收集的目的和范围，对用户密码采用BCrypt加密算法存储，对敏感数据的访问进行严格的权限控制，并承诺不向第三方泄露用户隐私。在数据保护方面，系统建立了定期数据备份机制，防止数据丢失。此外，本系统作为毕业设计项目，仅用于学术研究和学习目的，不涉及实际商业运营，因此不存在经营资质等法律风险。
3.5.3 经济可行性
从开发成本来看，本系统作为毕业设计项目，开发周期约为3个月，按每周投入20小时计算，总工作量约为240小时。参考市场上初级开发者的时薪标准（约50元/小时），开发人力成本约为12000元。开发过程中使用的IDE、数据库等工具均为免费版本或开源软件，无需额外的软件许可费用。从运营成本来看，若系统投入实际生产环境，需要租用云服务器进行部署。参考阿里云的价格，一台2核4GB内存的入门级云服务器年费约为1000元，MySQL云数据库年费约为1500元，Redis缓存服务年费约为800元，域名和SSL证书年费约为200元，合计年运营成本约为3500元。从收益角度分析，家政服务平台可通过向阿姨收取服务佣金（通常为订单金额的10%-20%）获得盈利。假设平台月均完成订单200单，平均客单价200元，按15%佣金比例计算，月收入约为6000元，年收入约为72000元，扣除运营成本后仍有可观的利润空间。综上所述，本项目开发投入较低，运营成本可控，具有良好的经济可行性。
3.6 本章小结
本章对贝贝家政服务系统进行了全面的需求分析。首先通过用例图和用例描述明确了普通用户、家政阿姨和系统管理员三类角色的功能需求，涵盖用户认证、浏览服务、预约服务、订单管理、个人中心、接单处理、工作管理、人员管理、服务管理和数据统计等十大核心模块。其次从可靠性、易用性、安全性和性能四个维度分析了系统的非功能性需求。最后从技术、法律和经济三个层面论证了项目的可行性，为后续的系统设计与实现奠定了基础。
